## 1. PRINCIPIOS ARQUITECTГ“NICOS DEL FRONTEND

### P1 вЂ” Agnosticismo de Motor
NingГєn motor (SchemaDesigner, LogicBridge, DocumentDesigner, AEE) conoce la implementaciГіn interna de otro. Solo se comunican a travГ©s de contratos de props y callbacks.

### P2 вЂ” Micro-Engines Desacoplados (MCA)
Las utilidades de UI (selectores de artefactos, paneles de propiedades, mapeadores) son componentes autГіnomos y reutilizables. Son invocados por cualquier motor que los necesite. No pertenecen a ningГєn motor especГ­fico.

### P3 вЂ” Cero LГіgica de Negocio en el Frontend
El frontend es una capa de proyecciГіn y captura. Toda computaciГіn, transformaciГіn y resoluciГіn de datos vive en el Backend (Logic Bridge в†’ `logic_engine.gs`).

### P4 вЂ” ModularizaciГіn por Responsabilidad
Cada motor principal tiene su propia carpeta con sub-artefactos. NingГєn archivo superarГЎ las 300 lГ­neas sin justificaciГіn documentada. Los hooks, paneles y utilidades son archivos separados.

## 3. MГ“DULOS DE SUPERFICIE (Lo que ve el usuario)

### 3.1 Core Connection
**Artefacto:** `shell/CoreConnectionView.jsx`

Permite al usuario vincular uno o mГєltiples Cores (instancias de Google Apps Script deployadas). Gestiona:
- URL del Core (endpoint del Web App GAS)

**Contrato:** Al conectar, el sistema popula el `app_state.js` con el `activeWorkspaceId` y el manifiesto de capacidades del Core.

---

### 3.2 Workspace Dashboard
**Vista principal de trabajo.** Se accede tras conectar el Core.

Proyecta los **pines del workspace activo** como tarjetas organizadas por `class`. Cada tarjeta expone las acciones que su ГЎtomo soporta segГєn `protocols`.

**Grupos de tarjetas en el Dashboard:**

#### Grupo A вЂ” Schemas (DATA_SCHEMA)
- BotГіn **+ Nuevo Schema** в†’ invoca `ATOM_CREATE` con `class: 'DATA_SCHEMA'` в†’ abre `schema_designer.jsx`
- Cada tarjeta: `[Abrir diseГ±ador]` `[Correr en AEE]` `[Conectar]`

#### Grupo B вЂ” Documentos (DOCUMENT)
- BotГіn **+ Nueva Plantilla** в†’ invoca `ATOM_CREATE` con `class: 'DOCUMENT'` в†’ abre `DocumentDesigner.jsx`
- Cada tarjeta: `[Abrir editor]` `[Previsualizar]` `[Conectar]`

#### Grupo C вЂ” Logic Bridges (WORKFLOW con `class: 'BRIDGE'`)
- BotГіn **+ Nuevo Bridge** в†’ crea el ГЎtomo y abre `BridgeDesigner.jsx`
- Cada tarjeta permite configurar **mГєltiples entradas (sources)** y **mГєltiples salidas (targets)**

#### Grupo D вЂ” Apps en ejecuciГіn
- BotГіn **+ Nueva App** в†’ selecciona un Schema + Bridge y los combina en una sesiГіn AEE
- Abre `AEE_Dashboard.jsx` como vista principal

### 3.3 Apps вЂ” AEE (Agnostic Execution Engine)
**Artefacto:** `macro_engines/AEE_Dashboard.jsx`

Corre un formulario (`DATA_SCHEMA`) en tiempo real, captura los valores ingresados por el usuario, los envГ­a al Logic Bridge asignado (`LOGIC_EXECUTE`), y proyecta el resultado.

**No contiene lГіgica de negocio.** Es un proyector reactivo de contratos.

Modos:
- **Form Mode:** Proyecta el schema como formulario interactivo (`FormRunner.jsx`)
- **Dashboard Mode:** Proyecta el resultado del Bridge en widgets configurables (`ResultPanel.jsx`)
