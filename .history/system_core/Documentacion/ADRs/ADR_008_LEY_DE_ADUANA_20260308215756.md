# ADR_008 — LEY DE ADUANA Y SINCERIDAD INFRAESTRUCTURAL

> **Versión:** 1.0
> **Estado:** VIGENTE — Mandato Radical
> **Alcance:** Todo Provider, Protocolo y Orquestador en Indra.

---

## 1. PROPÓSITO
Este ADR establece la **Ley de Aduana**. El sistema ha sufrido de entropía debido a la tolerancia ante datos mal formados. A partir de este momento, Indra adopta una política de **Tolerancia Cero** y **Fallo Ruidoso** para garantizar el determinismo radical.

---

## 2. EL AXIOMA DE LA ADUANA (Fail-Fast)
La Aduana (Middleware/Protocol Router) no es un traductor; es un **Filtro de Sinceridad**.

1. **PROHIBICIÓN DE TRADUCCIÓN:** La Aduana tiene estrictamente prohibido "arreglar" datos (ej. traducir `columns` a `fields`). Si un dato no viene en el formato canónico, es **Materia Oscura** y debe ser interceptado.
2. **FALLO RUIDOSO (Rally-to-Root):** Si un Provider viola el contrato, la Aduana debe lanzar una excepción `CONTRACT_VIOLATION` inmediata, bloqueando la ejecución y señalando al desarrollador exactamente qué Provider está fallando.
3. **RESPONSABILIDAD DEL PROVIDER:** La normalización es responsabilidad **exclusiva** del Provider en su origen. Si el dato es Notion, el Provider de Notion debe entregar `fields`. Si no lo hace, el Provider está roto.

---

## 3. CONTRATO CANÓNICO DE PAYLOAD (DATA_SCHEMA)
Para que la pipeline de vacío fluya, la estructura de un `DATA_SCHEMA` debe ser inmutable en todo el sistema:

```json
{
  "id": "ID_NATIVO",
  "class": "DATA_SCHEMA",
  "handle": {
    "ns": "com.indra...",
    "alias": "label_slug",
    "label": "Nombre Humano"
  },
  "payload": {
    "fields": [
      {
        "id": "field_id",
        "alias": "field_alias",
        "label": "Field Label",
        "type": "TEXT|MATH|FRAME|...",
        "children": [] 
      }
    ]
  }
}
```

---

## 4. PLAN DE ACCIÓN SISTÉMICO

### Fase 1: Blindaje de la Aduana (Backend)
- Modificar `protocol_router.gs` para que la validación sea coercitiva. Si un `DATA_SCHEMA` no tiene `payload.fields` como un array, la respuesta se bloquea con `CONTRACT_VIOLATION`.

### Fase 2: Saneamiento de Providers (La Raíz)
- Auditar `provider_system.js`, `provider_notion.js` y otros para asegurar que proyectan los datos al estándar antes de retornar.

### Fase 3: Purga de la UI (Agnosticismo)
- Eliminar de todos los componentes React (`PortManager`, `FieldMapper`, etc.) cualquier lógica de fallback (`||`). 
- Los componentes deben asumir que el dato es **Sincero**. Si el código dice `field.label` y no existe, el componente debe fallar. Esto fuerza la corrección del problema en la raíz, no en la rama.

---

## 5. CONSECUENCIAS
- **Positivas:** Determinismo total. El sistema "fluye" porque no hay dudas. La arquitectura de micro-contenedores (MCA) se vuelve verdaderamente intercambiable.
- **Negativas:** Los errores de desarrollo serán más frecuentes y ruidosos al principio, ya que el sistema dejará de perdonar la pereza estructural.

---

*La tolerancia es el combustible de la entropía. Indra solo respira Sinceridad.*
