# ADR_008 — LEY DE ADUANA Y SINCERIDAD INFRAESTRUCTURAL

> **Versión:** 2.0 — Sinceridad Referencial
> **Estado:** VIGENTE — Mandato Radical
> **Alcance:** Todo Provider, Protocolo, Orquestador y componente de UI en Indra.
> **Reemplaza:** v1.0 (Fail-Fast solo en salida de providers)

---

## 1. PROPÓSITO

Este ADR establece dos leyes complementarias que garantizan el **determinismo radical** del sistema:

1. **Ley de Aduana (Fail-Fast de Contrato):** Lo que entra al sistema debe ser canónico. Lo que sale del sistema debe ser canónico.
2. **Ley de Sinceridad Referencial (Portal de Sinceridad):** Lo que existe en el sistema debe reflejar la realidad física del almacenamiento. Las referencias no pueden mentir.

---

## 2. LEY DE ADUANA — FILTRO DE SINCERIDAD (v1.0, vigente)

### 2.1 El Axioma del Fail-Fast
La Aduana (`protocol_router.js`) no es un traductor; es un **Filtro de Sinceridad**.

- **PROHIBICIÓN DE TRADUCCIÓN:** Si un dato no viene en formato canónico, es **Materia Oscura** y se intercepta con `CONTRACT_VIOLATION`.
- **FALLO RUIDOSO:** El error señala exactamente qué Provider está fallando. No hay silencio.
- **RESPONSABILIDAD DEL PROVIDER:** La normalización es exclusiva del Provider en su origen.

### 2.2 Validación Coercitiva de Payload por Clase

| `class` | Payload mínimo garantizado en nacimiento |
|---------|------------------------------------------|
| `DATA_SCHEMA` | `payload.fields: []` (Array) |
| `WORKFLOW` | `payload.stations: []` (Array) |
| `BRIDGE` | `payload.operators: []` (Array) |

Si un Provider entrega un átomo de estas clases sin el campo mínimo, el router lanza `CONTRACT_VIOLATION` antes de retornar al cliente.

## 3. LEY DE SINCERIDAD REFERENCIAL — PORTAL DE SINCERIDAD (v2.0, nuevo)

### 3.1 El Problema que Resuelve

Un átomo puede ser referenciado como **Pin** en un Workspace. Si el átomo es borrado físicamente, la referencia queda "flotando" (Ghost Pin). El sistema anterior tenía un "Conserje Inteligente" (`_system_purgeAtomFromAllWorkspaces_`) que intentaba limpiar estas referencias tras el borrado mediante un escaneo masivo. Ese modelo es **anti-axiomático** porque:

- Crea dependencia entre operaciones ortogonales (borrar ≠ limpiar referencias).
- Si el escaneo falla, el sistema queda en estado inconsistente sin saberlo.
- Sobrecarga el módulo de infraestructura con "inteligencia relacional".

### 3.2 El Principio Arquitectónico

> **La Sinceridad vive en el momento del acceso, no en el momento del borrado.**

Un sistema de archivos (Drive) es la fuente de verdad. La existencia de un archivo es binaria: está o no está. El sistema no debe mantener un grafo de relaciones; debe **consultar la realidad en el momento que se necesita**.

### 3.3 El Axioma del Borrado Físico Determinista

```
ATOM_DELETE = Borrar el archivo. Fin.
```

El `ATOM_DELETE` **NO** propaga, **NO** escanea workspaces, **NO** purga referencias. Su único dharma es eliminar la materia física. Es una operación $O(1)$.

**Implicación:** Un Ghost Pin puede existir temporalmente. No es un estado de error; es un estado de **desincronización transitoria** que el Portal de Sinceridad resuelve en el momento de la lectura.

### 3.4 El Portal de Sinceridad (`_system_batchVerifyExistence_`)

Al momento en que el sistema lee una lista de referencias (pines), ejecuta una **verificación masiva de existencia física** usando `DriveApp.searchFiles()`. Este mecanismo:

- Es $O(1)$ en número de llamadas a Drive (una sola query independiente del número de pines).
- Marca cada referencia con `_orphan: true` si el archivo no existe o está en papelera.
- **No borra nada.** Solo proyecta la verdad.

```javascript
// Aplicado en: provider_system_workspace.js → _system_handlePinsRead
const existenceMap = _system_batchVerifyExistence_(pinIds);
const sincerePins = pins.map(pin => ({
    ...pin,
    _orphan: !existenceMap[pin.id]
}));
```

### 3.5 Homeostasis Voluntaria (`SYSTEM_WORKSPACE_REPAIR`)

El usuario (o un proceso explícito) puede solicitar la purga de los huérfanos detectados. Esta operación usa el mismo `_system_batchVerifyExistence_` y escribe el workspace saneado al disco. Es una operación **consciente y auditada**, no automática.

### 3.6 Proyección Visual del Estado (Frontend)

El flag `_orphan: true` es proyectado en `ArtifactCard.jsx`:
- La tarjeta se muestra con `opacity: 0.6` y badge `MATERIA_DESAPARECIDA`.
- El click para abrir el artefacto queda deshabilitado.
- El botón de `ATOM_DELETE` sigue disponible para permitir al usuario limpiar la referencia.

> **Axioma:** El caos se contiene a sí mismo cuando se hace visible. El usuario es el soberano.

## 4. DEBILIDADES CONOCIDAS Y DEUDA ABIERTA

### D1 — Ghost pins solo detectados en Workspace
El Portal de Sinceridad opera hoy **solo en la capa de Workspace** (`SYSTEM_PINS_READ`).

Las siguientes capas **no detectan** referencias muertas internamente:

| Capa | Referencia potencialmente muerta | Estado |
|------|----------------------------------|--------|
| `WorkflowDesigner` | `station.bridge_id` → Bridge borrado | ⚠️ Sin detección |
| `WorkflowDesigner` | `station.schema_id` → Schema borrado | ⚠️ Sin detección |
| `BridgeDesigner` | `config.relation_silo_id` → Schema borrado | ⚠️ Sin detección |
| `BridgeDesigner` | Puertos de entrada/salida → Schema borrado | ⚠️ Sin detección |

**Mitigación recomendada (Deuda D1):** Extender el Portal de Sinceridad a las capas de hidratación de cada motor (`useBridgeHydration`, `useWorkflowHydration`) para que detecten y señalicen dependencias muertas.

### D2 — `_system_batchVerifyExistence_` solo para provider `system`
La verificación masiva usa `DriveApp.searchFiles()`, que solo funciona para átomos nativos de Drive. Los átomos de providers externos (Notion, etc.) no pueden verificarse con este mecanismo.

**Mitigación recomendada (Deuda D2):** Cada provider externo debe implementar `ATOM_EXISTS` como protocolo ligero. El Portal de Sinceridad podría delegarle la verificación por provider.

### D3 — `_system_propagateNameChange` sigue siendo un escaneo
La función que propaga cambios de nombre en todos los workspaces sigue siendo $O(n_{workspaces})$ y usa el mismo patrón de "Conserje Inteligente" que fue eliminado del borrado.

**Mitigación recomendada (Deuda D3):** Los pines almacenan una copia del `handle` en el momento del anclaje. Si el label cambia, el pin mostrará el label desactualizado hasta que el usuario lo recargue (eventual consistency). Esto puede aceptarse como trade-off de agilidad o resolverse con el mismo Portal de Sinceridad ampliado.

### D4 — `identity_evolution.js` es código muerto
El módulo de reparación de identidades (`5_diagnostics/identity_evolution.js`) fue diseñado para "curar" átomos con identidades degradadas post-facto. Esto viola el Axioma del Borrado Físico Determinista: si el sistema es íntegro, no hay nada que curar.

**Acción:** Marcar como DEPRECATED. Si las reglas del sistema cambian (migración de contrato), se debe escribir un script de migración explícito y auditado, no una función de auto-reparación continua.

## 6. JERARQUÍA DE SINCERIDAD

```
ATOM_CREATE    → Nace sano o no nace (validación en provider_system_infrastructure)
ATOM_READ      → Lee la realidad sin modificarla
ATOM_UPDATE    → Actualiza la materia sin cambiar su identidad (id, class inmutables)
ATOM_DELETE    → Elimina la materia. La referencia es responsabilidad del Portal de Sinceridad.
SYSTEM_PINS_READ → Portal de Sinceridad: la verdad se resuelve en el acceso.
SYSTEM_WORKSPACE_REPAIR → Homeostasis voluntaria: el soberano decide cuándo sanear.
```
